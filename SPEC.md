# Shovelhead Atlas — 仕様書 v0.1

ショベルヘッドエンジンのインタラクティブ図解 × 知識集約アプリ。
このファイルをリポジトリ直下に置き、Claude Code CLI に「SPEC.md を読んでフェーズ0から始めて」と指示して開発を進める想定。

---

## 1. コンセプト

ショベルヘッド(1966–1984)のエンジン図をクリックすると、そのパーツの **歴史(年式変遷)・故障傾向・パーツ情報(OEM番号、代替品、入手先)** が、日英の情報源を横断した **日本語要約 + 出典リンク** として読める。

### 解決する課題

- 情報は世界中のフォーラム・ブログ・カタログに散在しているが、パーツ起点で統合されたものが存在しない
- 本場アメリカの情報(HDForums, Jockey Journal, XLForum 等)は言語の壁で日本のオーナーに届かない
- 「故障頻度・持病」の知識は経験者の頭とスレッドの中にしかなく、構造化されていない

### 設計上の最重要原則

- **OEMパーツナンバーが主キー。** ハーレーの部品番号(例: `26204-73`)は言語非依存の世界共通IDなので、日英の情報をここに束ねる。
- **転載ではなく「要約 + 出典リンク」。** マニュアルやカタログの図・本文は転載しない。図は自作SVG、テキストはLLM要約 + 必ず元ソースへのリンク。
- **データとアプリの分離。** 知識はリポジトリ内の JSON/Markdown。収集・翻訳・要約は Claude Code で回す ETL 運用。アプリは静的サイト。

---

## 2. スコープ

### MVP でやること

- 対象: **エンジンのみ・全年式(1966–1984)**。車体・足回り・電装(点火系除く)は対象外
- 年式差分をデータモデルの第一級市民として扱う(前期ジェネレーター 66–69 / 後期アルタネーター 70–84、コーンモーター移行などの変遷を表現できること)
- 2D SVG イラスト(エンジン側面図 → 系統別分解図の2階層)
- 静的サイトとして GitHub Pages で公開。公開リポジトリ

### MVP でやらないこと(将来構想)

- 3Dモデル、車体全体、ユーザー投稿機能、検索エンジン的な全文アーカイブ、多言語UI(まずUIは日本語、データに英語ソースリンク)

## 3. データモデル

`data/parts/*.json` に1パーツ1ファイル。スキーマ:

```json
{
  "id": "oil-pump-body",
  "names": { "ja": "オイルポンプボディ", "en": "Oil Pump Body" },
  "system": "lubrication",
  "variants": [
    {
      "years": [1968, 1980],
      "oem_numbers": ["26204-68", "26204-73"],
      "notes_ja": "68年に大型化。73年に…"
    }
  ],
  "history_ja": "…(年式変遷の要約。数段落)",
  "failure_modes": [
    {
      "title_ja": "シャフトシール抜けによる油圧低下",
      "years_affected": [1970, 1984],
      "severity": "high",
      "summary_ja": "…(症状・原因・対策の要約)",
      "sources": [
        { "url": "https://…", "title": "HDForums: …", "lang": "en", "accessed": "2026-07-18" }
      ]
    }
  ],
  "parts_availability_ja": "リプロはS&S/V-Twin MFGから…",
  "links": [
    { "url": "https://…", "title": "…", "lang": "en", "type": "forum" }
  ]
}
```

### データ品質ルール(Claude Code への恒久指示)

- **出典URLのない事実は書かない。** `failure_modes` と `history` の主張には必ず `sources` を付ける
- 要約は日本語、ソースは原語のままリンク。取得日(`accessed`)を記録
- 著作権物(マニュアルPDF、カタログ画像)のコピーはリポジトリに入れない

## 4. 図(SVG)

- **階層1:** エンジン側面図 `assets/svg/engine-side.svg` — 系統ごとのクリック領域
- **階層2:** 系統別分解図 `assets/svg/system-{system}.svg` — パーツごとのクリック領域
- クリック領域は `<g data-part-id="oil-pump-body">` で `data/parts` の `id` と紐付け
- **system 一覧(MVPの分解単位):**
  - `top-end` — ヘッド・ロッカーボックス・シリンダー・ピストン
  - `bottom-end` — クランクケース・フライホイール・コンロッド・ベアリング
  - `cam-drive` — カム・カムカバー・タペット・プッシュロッド
  - `lubrication` — オイルポンプ・オイルライン・ブリーザー
  - `carburetor` — キャブ(ケーヒン/S&S/ベンディックス等の変遷含む)
  - `ignition` — ポイント/電子進角・ジェネレーター/アルタネーターの移行
- SVG は Claude Code に描かせてよいが、**正確さより「どこがどのパーツか分かる図式的イラスト」を優先**。実写トレースは著作権的に自分で撮った写真のみ可

---

## 5. アプリ(静的サイト)

- フレームワークなしの **素の HTML + CSS + JS**(ビルド不要、GitHub Pages 直配信、長期メンテ最小)
- `index.html` が側面図を表示 → 系統クリックで分解図 → パーツクリックで右パネル(またはモバイルはボトムシート)に JSON の内容をレンダリング
- **URL 設計:** `#/{system}/{part-id}` のハッシュルーティング(パーツへの直リンク共有可)
- **年式フィルタ UI:** 年式スライダー(1966–1984)で、その年式に存在する variant のみ強調表示。デフォルトは後期(70–84)ビュー

---

## 6. データ収集の運用(Claude Code ETL)

`.claude/commands/research-part.md` としてスラッシュコマンドを作る:

```
/research-part <part-id>

1. data/parts/<part-id>.json を読む(なければ雛形作成)
2. WebSearch で英語圏(HDForums, Jockey Journal, XLForum, CAI blog等)と
   日本語圏(バージンハーレー、ショップブログ等)を検索
3. 見つけた情報を history / failure_modes / links に反映(必ず sources 付き)
4. 差分をユーザーに提示してからコミット
```

- 週次や気が向いたときに `/research-part` を回してデータを育てる
- 情報の正誤に自信がない場合は `"confidence": "low"` を付けて残す(消さない)

## 7. フェーズ計画

- **Phase 0: 足場** — リポジトリ初期化、ディレクトリ構成、スキーマのJSON Schema化、GitHub Pages 配信、side-view SVG のラフ、パーツ3個分のダミーデータで動線確認
- **Phase 1: lubrication 系統を完成させる** — オイルポンプ周りは持病情報が濃く、パーツ数が少ないので縦に1本通すのに最適。分解図SVG + 全パーツのデータを `/research-part` で埋める
- **Phase 2: top-end / cam-drive** — ショベル最大の情報量ゾーン(バルブ周り・カム)
- **Phase 3: bottom-end / carburetor / ignition** — 年式変遷が激しいゾーン。年式フィルタUIをここで本格化
- **Phase 4以降(構想)** — 車体系への拡張、英語UI、他機種(パン・エボ)テンプレート化

## 8. リスクと割り切り

- **著作権:** 図は自作、テキストは要約 + 出典。マニュアル類のホスティングはしない。ハーレー社の商標(ロゴ等)は使わず、名称は記述的使用に留める
- **故障頻度の信頼性:** フォーラム発の伝聞が混ざるため、頻度の定量化はしない。「報告が多い/散見される」レベルの表現 + `severity` + 出典数で示す
- **完璧主義の罠:** データが薄いパーツも公開してよい。「出典リンクだけある状態」も価値がある。育てるリポジトリとして運用する
